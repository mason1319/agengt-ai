# 星伴英语（StarMate English）第一阶段 PRD（v1.0）

**项目**：星伴英语（StarMate English）  
**阶段**：Phase 1（内部可用 Web 版）  
**目标**：交付第一阶段可运行闭环（非对外商用）  
**编写日期**：2026-06-06  
**状态**：待确认（第一阶段冻结前版本）

## 1. 项目定义

第一阶段只服务本机构内部运营，围绕“学生学习闭环 + 老师干预闭环 + 创始人对账闭环 + 外部咨询转化闭环”建立可落地系统。  

不做：

- 对外注册、对外商用多机构租户、在线支付闭环、原生 App/商用小程序；
- 复杂课程导学体系、完整营销体系、长期会员体系；
- 数据库重构和字段级建模。  

本阶段不启动 UI 设计，不编写前端代码，不设计数据表，仅产出可执行的产品与架构文档与上线执行约束。

## 2. 业务问题与成功目标

### 2.1 核心问题

- 创始人无法快速看穿咨询转化、课时消耗、AI 使用和老师处理异常；
- 老师缺乏异常处理闭环，重复人工沟通频繁；
- 家长看不到孩子课时变动原因，信任成本高；
- 学生学习任务反馈链路不透明，完成率低；
- 外部咨询提交与内训系统未打通，转化慢。

### 2.2 第一阶段成功目标（必须可验收）

1. 创始人：可在统一驾驶舱看到课程、课时、AI 用量、咨询转化和未处理异常；
2. 老师：只处理授权学生，能完成点名、异常记录与干预提交；
3. 家长：可查看子女课时消耗、学习摘要、课程安排；
4. 学生：可查看今日任务并在学习后看到反馈；
5. 外部咨询用户：可完成咨询提交、课程建议、试听预约，形成可跟进线索；
6. 技术交付：关键链路在 `/api/v1` 下可自测通过。

## 3. 用户角色与需求

### 3.1 创始人（Founder）

- 目标：1）看见机构状态，2）识别风险，3）对账可追溯。  
- 需求：
  - 经营驾驶舱（AI 用量、异常、课时风险、咨询漏斗）
  - 创始人课程/课时/缴费总览
  - 线索状态管理与老师处理可追踪

### 3.2 老师（Teacher）

- 目标：降低重复判断与重复通知成本。  
- 需求：
  - 授权学生列表与课程列表
  - 异常记录、点名、到课/请假/缺课处理
  - 学生详情与干预提交

### 3.3 家长（Parent）

- 目标：理解孩子是否正常学习，减少不必要沟通。  
- 需求：
  - 绑定孩子列表
  - 课时账户余额、扣减原因、缺课保留说明
  - 孩子课程与阶段学习摘要

### 3.4 学生（Student）

- 目标：有明确、可完成、可反馈的学习路径。  
- 需求：
  - 今日任务
  - 口语练习（第一阶段支持文本化可落地评估）
  - 复盘/结果查看
  - 课程安排可见

### 3.5 外部咨询用户

- 目标：快速提交咨询并进入到试听/跟进通道。  
- 需求：
  - 咨询信息与课程偏好收集
  - 课程价格/名额/试听时间公开展示
  - AI/规则化咨询回复与线索状态更新

## 4. 场景与用户旅程（第一阶段）

### 场景 A：学生学习

学生登录 → 查看今日路径 → 完成学习任务 → 提交/复盘 → 接收反馈并记录  

### 场景 B：老师处理

老师登录 → 查询今日异常/授权学生 → 提交点名与干预 → 干预可被创始人复核  

### 场景 C：家长知情

家长登录 → 选择绑定孩子 → 查看学习摘要与课时明细 → 关注课程安排和扣减原因  

### 场景 D：创始人监控

创始人登录 → 查看驾驶舱 → 跟踪未处理异常 → 核对课时与线索状态 → 导出对账信息  

### 场景 E：咨询转化

咨询用户提交 → AI/规则化建议 → 预约试听 → 线索转化与接管

## 5. 功能范围与优先级（第一阶段）

### 5.1 P0（上线前必须）

- 登录与鉴权（角色登录、JWT、RBAC）
- `/api/v1` 统一接口约束
- 学生：今日任务、复盘、口语反馈
- 老师：学生授权、异常列表、干预提交
- 家长：课时可见、课程清单、孩子摘要
- 创始人：驾驶舱、课程和咨询状态汇总
- 公共：课程列表、咨询提交、试听预约、线索 AI 回复
- 语音/素材：R2 上传与签名下载
- 生产链路：Cloudflare Pages + Functions + D1 + R2

### 5.2 P1（建议下周内完成）

- 公开课程详情细化
- 创始人课程与课时批量筛选
- 外部咨询接管流转增强
- 导出端点与轻量报表

### 5.3 本阶段明确不做

- 在线支付、计费开立订单流；
- 公开注册；
- 商业版多机构平台能力；
- 复杂 AI 大模型自动化策略（先用规则化/Mock fallback）。

## 6. 接口与交付规范（对开发的约束）

1. 所有业务接口统一 `/api/v1` 前缀；
2. 返回体统一 `code/message/data/request_id/trace_id`；
3. 非公开接口全部鉴权，且角色隔离必须在服务端兜底；
4. 无真实后端动作的控件、字段、状态不可上线；
5. 所有写操作记录来源与原因；
6. AI 失败场景必须给明确降级提示。

## 7. 非功能需求

- 性能：核心列表 P95 ≤ 800ms（Cloudflare 边缘自测）；
- 可用性：移动端 375px 无横向溢出；
- 可观测：所有核心接口带 `request_id` 与 `trace_id`；
- 安全：R2 默认私有、下载签名；PII 最小化、脱敏展示；
- 稳定：关键接口可用率目标阶段内保持 99%以上；
- 合规：咨询信息需有隐私同意字段。

## 8. 接口验收指标

- `code=0` 成功率（核心闭环 API）连续 5 次验收不低于 99%；
- `/api/v1/student/today-path`、`/api/v1/teacher/students`、`/api/v1/parent/children`、`/api/v1/founder/cockpit`、`/api/v1/public/leads` 等关键接口可正常调用；
- 创建/修改/点名/干预/线索流转均有审计记录；
- 公开咨询链路 30 秒内可提交成功。

## 9. 风险与开放问题

### 风险

- AI 供应商/模型不可用：采用规则化降级；
- 多环境代理导致本地回环访问失败：启动链路使用固定 `NO_PROXY`；
- 课时与请假策略在组织内部边界差异较大，需要统一规则。

### 开放问题（需确认）

1. 无故缺课是否计课时由谁最终决策；
2. 外部咨询是否必须强制绑定课程创建；
3. 课程公开信息的财务字段边界；
4. 创始人驾驶舱指标口径和刷新频率。
