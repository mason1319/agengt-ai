# 星伴英语（StarMate English）产品需求文档 v4.1

**版本定位**：内部自用执行版  
**项目类型**：本机构 AI 英语学习系统  
**状态**：执行版，作为第一阶段开发依据  
**日期**：2026-06-06  
**当前范围**：服务本机构创始人、老师、学生、家长；允许外部客户仅做 AI 咨询、课程查看、试听预约  
**暂不范围**：对外 SaaS、公开注册、在线支付、支付套餐、平台代理、App、小程序商用发布  

---

## 0. v4.1 核心结论

v4.1 不再按“对外售卖 SaaS”设计第一版，而是压缩成一个本机构能真实使用的 AI 英语学习系统。

产品总原则：**一切设计围绕解决问题，方便你、我、他。**

| 对象 | 要解决的问题 | 设计方向 |
|---|---|---|
| 你：创始人/机构 | 不想天天亲自盯学生、老师、招生、收费 | 看板化、线索化、异常化、收费记录清晰 |
| 我：老师/运营 | 不想重复沟通、重复统计、重复催促 | AI 客服先接待、系统自动归纳、老师只处理异常 |
| 他：学生/家长/客户 | 不知道怎么学、不知道报什么课、不知道孩子有没有进步 | 少选择、快反馈、课程价格清楚、试听入口清楚、家长报告清楚 |

任何新增功能必须回答三个问题：

| 问题 | 不满足时处理 |
|---|---|
| 解决谁的问题？ | 不做 |
| 是否更方便？ | 不做 |
| 是否能落到 API、数据和流程？ | 不做 |

第一阶段必须先证明七件事：

| 目标 | 验收标准 |
|---|---|
| 学生能学 | 学生登录后能看到今日任务、完成口语/复习任务、看到反馈 |
| 老师能管 | 老师只能看授权学生，能处理异常、提交干预 |
| 家长能懂 | 家长只能看自己孩子，能看到本周进步、待加强、剩余任务 |
| 创始人能控 | 创始人能看到学生活跃、老师处理、AI 用量、风险提醒 |
| 招生能接 | 新客户能通过 AI 客服了解课程、价格、时间并预约试听 |
| 课程能卖 | 课程表、开课时间、适合年级、班型、费用、名额能清楚展示 |
| 课时能清 | 缴费、剩余课时、上课消课、请假保留、缺勤处理能一目了然 |

第一阶段不追求功能大而全，不做完整商业化，不做 App，不做小程序。先用 Web/H5/PWA 让本机构跑起来。

## 1. 产品定位

### 1.1 一句话定位

星伴英语是本机构内部使用的 AI 英语学习系统，用 AI 帮学生完成口语、复习、任务反馈，用数据帮老师和创始人减少重复管理。

### 1.2 用户优先级

| 优先级 | 角色 | 第一阶段目标 |
|---|---|---|
| P0 | 学生 | 每天完成 AI 学习任务，获得即时反馈 |
| P0 | 老师 | 只处理异常学生，不做全量重复批改 |
| P0 | 创始人 | 看学习质量、老师处理、AI 成本和风险 |
| P1 | 家长 | 看懂孩子有没有进步，不参与复杂教学 |
| P1 | 外部咨询客户 | 看清课程、费用、时间，快速预约试听 |

### 1.3 当前取消内容

| 模块 | 第一阶段处理 |
|---|---|
| 对外 SaaS 套餐 | 不做 |
| 月付/年付/支付 | 不做 |
| 公开注册 | 不做 |
| 多机构平台招商 | 不做 |
| iOS/Android 原生 App | 不做 |
| 微信小程序正式版 | 不做，先用 H5/PWA |
| 文化墙完整运营 | 后置，只保留素材上传预留 |
| 真实大模型全面接入 | 可配置预留，第一阶段优先打通一个国内供应商或规则化降级 |

### 1.3.1 招生咨询与后期个人使用预留

本机构需要招生，因此 v4.1 增加“招生咨询与试听转化”模块。该模块不是公开注册，也不是对外 SaaS 售卖。外部家长或个人只能提交咨询、预约试听、领取测评建议，不能直接进入正式学习系统。

| 场景 | 当前处理 | 后期预留 |
|---|---|---|
| 本机构招生 | AI 客服接待、咨询表单、试听预约、线索跟进 | 可接入落地页、投放来源、转化分析 |
| 外部家长咨询 | AI 客服收集孩子年龄、年级、英语问题、联系方式 | 可增加自动测评建议和更完整对话 |
| 个人用户试用 | 暂不开放自助注册 | 后期可做个人体验账号、限额试用、付费升级 |
| 机构外部商用 | 暂不开放 | 本机构验证稳定后再做 SaaS 化 |

招生模块设计原则：

| 原则 | 说明 |
|---|---|
| 低门槛 | 家长 30 秒内能提交咨询 |
| 强转化 | 咨询后进入“待联系→已联系→试听预约→试听完成→转正式学生” |
| 不越权 | 咨询用户不能访问学生系统 |
| 不骚扰 | 必须有隐私告知和联系方式授权 |
| 可追踪 | 每条线索记录来源、跟进人、状态和下一步 |

AI 客服边界：

| 允许 | 禁止 |
|---|---|
| 回答课程方向、适合年龄、学习方式、试听流程 | 承诺提分、承诺录取、夸大效果 |
| 收集孩子年龄、年级、当前英语问题 | 收集无关敏感信息 |
| 推荐试听时间和课程方向 | 自动收费或自动开正式账号 |
| 生成家长可读的初步建议 | 替代老师做最终诊断 |
| 将对话摘要写入线索记录 | 向外部家长泄露内部学生案例 |

### 1.3.2 课程表、开课时间与收费记录

参考现有暑假课程表，v4.1 增加“课程排期与内部收费记录”能力。它用于招生展示、内部排课、家长查看自己孩子课程、创始人核对已收费，不等同于在线支付系统。

| 使用场景 | 展示内容 | 权限 |
|---|---|---|
| 公开招生页 | 课程名称、时间段、适合年级、课次、课程单价、连报价、限招人数、剩余名额、试听入口 | 未登录可见，不显示已收费名单和学生名单 |
| AI 客服 | 根据家长咨询推荐适合课程、试听时间、课程价格和连报价 | 未登录可用，只返回公开课程价格字段 |
| 家长端 | 自己孩子已报课程、上课时间、开课/结课时间、剩余课次、课程价格、缴费状态 | 仅绑定孩子可见 |
| 学生端 | 下一节课、今日课程、课程任务 | 仅本人可见 |
| 老师端 | 自己负责课程、上课时间、学生名单、上课记录 | 仅授权课程/学生可见，不默认显示收费 |
| 创始人/运营端 | 全部课程表、开课时间、报名人数、限招人数、收费状态、已收/未收、转正式学生 | 内部管理可见 |

课程表必须支持的核心字段：

| 字段 | 说明 | 公开展示 |
|---|---|---|
| `course_name` | 课程名称 | 是 |
| `course_type` | 课程类型：音标/自然拼读/课内同步/新概念/专项提升 | 是 |
| `class_type` | 班型：小班课/大班课/一对一 | 是 |
| `grade_range` | 适合年级：一年级、二年级、小升初等 | 是 |
| `start_date` | 开课日期 | 是 |
| `end_date` | 结课日期 | 是 |
| `weekday` | 上课星期 | 是 |
| `time_slot` | 上课时间段，如上午 8:30-10:30 | 是 |
| `total_sessions` | 总课次 | 是 |
| `session_duration` | 单次课时长 | 是 |
| `single_period_price` | 单期价格 | 是 |
| `bundle_price` | 连报/组合价格 | 是 |
| `capacity` | 限招人数 | 是 |
| `remaining_seats` | 剩余名额 | 是 |
| `teacher_id` | 授课老师 | 可选公开 |
| `classroom` | 教室/线上链接 | 内部或报名后可见 |
| `enrolled_count` | 已报名人数 | 内部可见，公开可显示剩余名额 |

班型规则：

| 班型 | 适用场景 | 人数建议 | 价格展示 |
|---|---|---:|---|
| 小班课 | 常规提升、同步课、自然拼读 | 6-15 人 | 公开单期价/连报价 |
| 大班课 | 暑假集训、公开体验、系统复习 | 16-40 人 | 公开单期价/连报价 |
| 一对一 | 个性化补弱、口语纠音、试听转化 | 1 人 | 可公开起步价或“咨询后报价” |

AI 客服推荐课程时，必须结合：年级、基础、目标、可上课时间、预算/价格接受度、班型偏好。

设计位置建议：

| 位置 | 放什么 |
|---|---|
| 公开咨询页 | 放“热门课程表”模块，卡片化展示课程价格、连报价、剩余名额 |
| AI 客服对话旁 | 放“AI 推荐课程”小卡片，展示价格和试听入口 |
| 创始人端 | 新增“课程与收费”页面，适合表格和筛选 |
| 家长端 | 成长卡下方放“孩子课程安排” |
| 老师端 | 老师首页放“今日/本周课程” |
| 学生端 | 学生首页放“下一节课” |

内部收费规则：

| 规则 | 说明 |
|---|---|
| 只做记录 | 第一阶段只记录线下/人工已收费，不接在线支付 |
| 公开价格 | 公开展示课程价格、连报价、课次、剩余名额，便于招生转化 |
| 不公开个人财务 | 公开课程表不显示某个学生是否缴费、已收金额、欠费状态 |
| 可对账 | 创始人可按课程、学生、时间筛选已收/未收 |
| 可追踪 | 收费记录写入审计 |
| 可转化 | 招生线索试听后可转报名并生成收费记录 |

### 1.3.3 家长缴费、学生课时与到课打卡

v4.1 增加“课时账户”能力，解决家长看不清缴费、学生剩余课时混乱、老师点名消课不清楚、缺课是否保留争议的问题。

| 场景 | 系统处理 |
|---|---|
| 家长缴费 | 家长端展示自己孩子的课程、已缴金额、缴费状态、购买课时 |
| 学生课时 | 每个学生有课时账户，展示总课时、已上课时、剩余课时、保留课时 |
| 每天打卡 | 老师上课时点名：到课、请假、缺课、迟到 |
| 正常到课 | 记录上课并消耗 1 次或对应课时 |
| 请假/未到但允许保留 | 不消课，课时保留到下次 |
| 缺课且按规则消课 | 按机构规则消耗课时，并显示原因 |
| 家长查看 | 家长能看到每次上课记录、扣课/保留原因 |
| 创始人查看 | 可按学生、课程、老师查看课时消耗和剩余 |

课时规则默认建议：

| 类型 | 是否消课 | 说明 |
|---|---|---|
| 到课 | 是 | 正常上课后消课 |
| 迟到 | 是 | 默认消课，可备注 |
| 请假 | 否 | 课时保留，下次再上 |
| 老师/机构原因停课 | 否 | 课时保留 |
| 无故缺课 | 待确认 | 可由创始人配置是否消课 |
| 补课 | 是 | 补课完成后消耗保留课时 |

设计目标：家长、老师、创始人都能一眼看懂“钱交了多少、课上了多少、还剩多少、为什么扣/没扣”。

### 1.4 行为心理设计原则：简单、高效、上头

本产品不是做“功能很多的系统”，而是围绕真实用户的人性弱点降低学习阻力。文档内部可用“低情商、基础弱、贪快、懒启动、馋反馈、内卷、内耗”描述问题，但用户可见文案必须专业、中性、鼓励，不允许羞辱学生、老师或家长。

| 用户特性/场景 | 本质问题 | 产品机制 | 第一阶段落地 |
|---|---|---|---|
| 低情商 | 不会表达，沟通容易有情绪 | AI 生成中性话术，减少老师/家长冲突 | 家长摘要、老师干预备注支持 AI 话术 |
| 基础弱 | 看不懂复杂解释，容易放弃 | 错点拆成最小动作，给示范和重试 | 口语错误词、复习任务、下一步建议 |
| 懒启动 | 不愿开始，不想做复杂选择 | 每天只给少量明确任务，默认下一步 | 今日 1-3 个任务，主按钮“开始” |
| 贪快 | 想马上看到结果 | 即时反馈、即时评分、即时进度 | 口语提交后给得分、错误点、建议 |
| 馋反馈 | 需要持续奖励才愿意坚持 | 连续学习、徽章、XP、成长提示 | streak、完成反馈、结果页正反馈 |
| 内卷 | 总和别人比，焦虑变高 | 只和自己比，不做公开排名 | 家长端不显示其他孩子；学生端只显示个人进步 |
| 内耗 | 老师、家长、创始人反复沟通 | 系统自动整理异常和处理状态 | 老师异常看板、创始人 SLA、家长周报 |

设计硬规则：

| 规则 | 说明 |
|---|---|
| 每页一个主操作 | 学生页不能堆多个同级按钮 |
| 每次只推下一步 | 不让学生自己规划学习路径 |
| 反馈必须即时 | 做完任务立刻有结果或明确等待状态 |
| 结果必须具体 | 不说“继续努力”，要说“哪个词/哪个音/哪个知识点” |
| 激励不过度 | 不做赌博式随机奖励，不做公开排名，不制造未成年人焦虑 |
| 老师只处理异常 | 不让老师重新做全量批改 |

## 3. 核心业务闭环

### 3.1 学生学习闭环

```text
登录 → 今日任务 → 口语练习/复习任务 → 提交 → AI/规则反馈 → 生成错题/复习 → 路径更新
```

验收标准：

| 节点 | 必须结果 |
|---|---|
| 今日任务 | 从 D1 返回，不允许硬编码 mock |
| 口语练习 | 录音上传 R2，生成一次评分或规则化反馈 |
| 复习任务 | 错题/单词进入复习队列 |
| 结果页 | 展示得分、错误点、下一步建议 |
| 学习事件 | `learning_events` 必须落库 |

### 3.2 老师干预闭环

```text
系统发现异常 → 老师查看授权学生 → 确认问题 → 提交干预 → 记录处理结果
```

异常触发：

| 条件 | 动作 |
|---|---|
| 连续 3 天未完成任务 | 标记低活跃 |
| 口语连续 3 次低分 | 标记口语风险 |
| 复习任务连续错误 | 标记薄弱知识点 |
| 家长申请人工关注 | 进入老师待处理 |

### 3.3 家长反馈闭环

```text
学生完成任务 → 系统生成成长摘要 → 家长查看 → 必要时申请老师关注
```

家长只看四类信息：

| 信息 | 说明 |
|---|---|
| 本周学了什么 | 学习任务摘要 |
| 哪里进步 | 具体能力维度 |
| 哪里要加强 | 最多展示 3 个重点 |
| 接下来怎么做 | 下一周任务建议 |

### 3.4 创始人管理闭环

```text
看板 → 风险提醒 → 老师处理进度 → AI 用量/成本 → 决策
```

创始人第一阶段只看内部经营与教学质量，不看套餐销售。

### 3.5 招生咨询闭环

```text
外部家长/个人访问咨询页
  → AI 客服自动回复并收集孩子情况和联系方式
  → 系统生成咨询线索
  → AI 客服推荐试听方向和可预约时间
  → 家长确认预约
  → 运营/老师必要时接管
  → 试听完成
  → 转为正式学生并绑定家长
```

线索状态：

| 状态 | 说明 |
|---|---|
| `new` | 新咨询 |
| `contacted` | 已联系 |
| `trial_scheduled` | 已预约试听 |
| `trial_completed` | 已完成试听 |
| `enrolled` | 已转正式学生 |
| `invalid` | 无效线索 |
| `archived` | 已归档 |

招生咨询第一阶段只做 AI 客服接待、线索、预约和跟进，不做在线支付、公开注册和自动开课。

## 6. AI 与智能体范围

### 6.1 第一阶段智能体

| 智能体 | 第一阶段做什么 |
|---|---|
| StarVoice 星语官 | 口语练习、录音评分、错误反馈 |
| StarPath 星路官 | 根据错误生成下一步任务 |
| StarHome 星家官 | 把学习数据转成家长看得懂的摘要 |
| StarOps 星账官 | 内部 AI 用量、异常和成本提醒 |

### 6.2 第一阶段 AI 真实度要求

第一阶段不允许出现“空按钮”和“静态假结果”。AI 能力分两档：

| 档位 | 说明 | 验收 |
|---|---|---|
| 规则化 AI | 根据任务、答案、录音提交状态生成反馈 | 必须可落库、可追踪、可复查 |
| 国内模型接入 | 讯飞/通义/Kimi 等可配置接入 | 如果密钥可用，优先接真实接口 |

正式验收不接受纯前端写死反馈。即使是规则化结果，也必须由后端接口生成并落库。

### 6.3 AI 降级

| 场景 | 降级 |
|---|---|
| 模型不可用 | 使用规则化反馈 |
| 语音评分失败 | 保存录音，提示稍后重试 |
| 超过内部预算 | 限制真实模型调用，保留基础任务 |
| 内容安全风险 | 不展示给学生，进入老师/创始人复核 |

样式规则修订：允许一个全局 CSS 入口承载 Tailwind 和 shadcn token，禁止业务页面自建 CSS 文件。

### 10.1 国际化研发标准补充

第一阶段技术栈必须按“接口先行、类型安全、可审计、可部署、可删冗余”的标准执行。代码质量不以页面是否能打开为唯一标准，必须同时满足前端、后端、数据、权限、扫描和验收要求。

| 层 | 强制要求 |
|---|---|
| 语言 | 前端和后端统一 TypeScript，开启 strict，不允许用 `any` 逃避类型 |
| 前端框架 | React + Vite + TypeScript，按 feature 分层，不允许页面里堆业务杂代码 |
| UI 组件 | Figma 定稿后使用 shadcn/ui 官方组件和 Tailwind token，不新增零散自定义 CSS 文件 |
| API 框架 | Cloudflare Pages Functions，建议统一使用 Hono 风格路由或原生 Functions 二选一，不允许混杂 |
| 接口契约 | OpenAPI 3.1 作为接口源头，页面控件、API、D1 字段必须一一映射 |
| 入参校验 | 所有写接口必须做 Zod 或等价 schema 校验 |
| 数据库 | Cloudflare D1，所有表和字段通过 `schema.sql` 和 migration 管理 |
| 文件存储 | Cloudflare R2，语音、图片、资料使用私有桶和签名 URL |
| 权限 | 统一 RBAC 中间件，老师、家长、学生、创始人、公开咨询权限分层 |
| 日志审计 | 登录、AI 调用、录音评分、收费、课时调整、点名、人工接管必须写审计 |
| AI | 国内模型适配层优先，模型失败时走后端规则化降级，不允许前端伪造 AI 结果 |

### 10.2 禁止冗余和无用代码

| 禁止项 | 处理 |
|---|---|
| 无 API 的按钮、弹窗、输入框、统计卡 | 删除或补接口，不允许保留 |
| 前端硬编码业务 Mock 数据 | 删除，改为 D1/API 返回 |
| 未使用组件、未使用 hooks、未使用工具函数 | 删除 |
| 重复组件、重复请求封装、重复类型定义 | 合并到统一模块 |
| 页面内堆叠大段业务逻辑 | 拆到 feature service、API client、hooks |
| `console.log`、临时 TODO、调试入口 | 交付前删除 |
| 未脱敏展示手机号、家长联系方式 | 不允许上线 |
| 绕过权限直接查表或直接返回全量数据 | 不允许上线 |

| 控件 | 每个按钮、弹窗、输入框、统计卡必须绑定 API 或明确路由 |
| 数据 | 页面展示数据必须来自 API，不允许业务假数据 |
| 状态 | Loading、Empty、Error、Success 状态完整 |
| 权限 | 路由和组件都按角色限制，不显示越权入口 |
| 移动端 | 375px 不横向溢出，手机端主流程可完成 |
| 可访问性 | 表单 label、焦点态、颜色对比达到基础 WCAG 要求 |
| 性能 | 首屏不加载无关大包，图片和音频按需加载 |
| 结构 | 页面只负责组合，业务逻辑进入 feature/service 层 |

### 10.5 后端验收标准

| 项 | 必须达到 |
|---|---|
| 鉴权 | 除公开课程和咨询外，所有 API 必须校验 token |
| 授权 | 老师只能访问授权学生，家长只能访问自己孩子，学生只能访问自己数据 |
| 入参 | 所有写接口必须 schema 校验，错误返回统一格式 |
| 事务 | 收费、报名、点名、消课、保留课时必须保证数据一致 |
| 审计 | 关键写操作必须记录操作者、时间、对象、前后状态和原因 |
| 限流 | 公开咨询、AI 回复、录音上传必须限流 |
| 隐私 | 家长联系方式脱敏，R2 文件不裸公开 |
| 可观测 | API 错误、AI 失败、R2 上传失败、D1 写入失败必须可追踪 |

## 11. 安全与合规

### 11.1 儿童与家长授权

本系统面向 4-16 岁学生。第一阶段必须内置家长授权，不再把 13 岁以下合规后置。

| 要求 | 第一阶段处理 |
|---|---|
| 儿童账号 | 由创始人/老师创建，绑定家长 |
| 家长授权 | 家长首次登录确认授权 |
| 语音采集 | 明示用途：口语练习和评分 |
| 数据删除 | 创始人可申请删除学生数据 |
| 数据导出 | 第一阶段先提供后台导出预留 |
| 社交功能 | 不做 |
| 排行榜 | 不做 |

### 11.2 安全基线

| 项 | 要求 |
|---|---|
| 登录 | 失败限频，token 过期处理 |
| 权限 | 所有接口校验 role + institution_id |
| 文件上传 | 限制类型、大小、R2 私有访问 |
| 审计 | 写操作必须有 `trace_id` |
| PII | 家长手机号脱敏 |
| 备份 | D1 导出和 R2 备份策略预留 |
