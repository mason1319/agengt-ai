# StarMate English 第一阶段：跨端落地路线（H5 / iOS / Android / 小程序）

> 当前原型是 Web 先行方案。该文档只给“后续不重写”的改造路径。
> 具体操作命令见：`docs/phase1-cross-platform-ops.md`

## 1. 目标

- 一套业务前端可同时服务：
  - H5 手机网页（当前阶段）
  - iOS App（原生壳）
  - Android App（原生壳）
  - 微信小程序（过渡期 H5 或原生小程序框架重写）
- 仍保持：  
  - 角色切换（创始人 / 老师 / 家长 / 学生 / 平台）
  - 套餐与额度规则
  - 五级权限与脱敏策略
  - 先模拟输出、后接真实 AI

## 2. 当前阶段建议（不改现有页面）

1. 保持 `React + Vite` 原型作为统一“展示端内核”。
2. 所有业务动作只依赖前端状态与模拟数据，未来替换为 API 时不改界面树。
3. 所有交互统一从“按钮动作 -> 状态反馈”闭环，方便后续把动作映射到服务接口。

## 3. iOS / Android（Capacitor 路线）

- 统一 Web 产物发布：`npm run build`
- 用 Capacitor 装配双端壳（后置）：
  - 初始化脚本：`npm run mobile:bootstrap`
  - 先决条件：网页验收通过
- 壳内配置项只处理：
  - 启动页 / 状态栏样式
  - 网络白名单
  - 设备权限（后续）
- 优点：
  - 前端组件与逻辑不用重写
  - 后续可按需逐步提取原生能力（推送、文件/相机、通讯录）
- 注意：
  - 先做「真机可触控」和「横向溢出」再做原生能力接入。

## 4. 微信小程序方案

- 阶段 A（最快）：H5 直入小程序 Webview（验收业务与流程）；
- 阶段 B（稳定）：抽离统一接口层后，迁移到 Taro / uni-app；
- 阶段 C：将平台运营、教师、家长、学生页面按能力粒度迁移成小程序组件（可复用文案与规则配置）。

## 5. H5 手机网页强化

- 已在当前样式中完成第一层适配：响应式布局 + 断点 1180/720。
- 下一步建议补齐：
  - PWA 与离线缓存（课程卡片、素材说明）
  - 输入框/按钮触区放大（移动端可点率）
  - 统一 Safe Area 适配（适配刘海屏）
  - 页面级骨架与加载状态

## 6. 方便后续修改的结构策略（关键）

建议把以下配置抽离为独立文件，减少后续平台改造成本：

1. `role-config`：角色菜单、图标、视图入口文案  
2. `plan-config`：套餐额度、计费口径、到期动作  
3. `permission-config`：角色-权限映射  
4. `mock-vs-api`：接口开关（`模拟` <-> `真实`）

当前第一阶段可以把这些作为常量放在独立文件，后续改成 JSON/服务端下发都不需大改组件。

## 7. 发布与验收口径（多端统一）

- 同步验收点：
  - 首页可见且不横向溢出
  - 角色切换可见且按钮可点
  - 老师端联系方式脱敏规则可见
  - 平台端到期策略与套餐状态可读可变更
- 阶段验收以“可操作闭环”为准，不以代码行数为准。
